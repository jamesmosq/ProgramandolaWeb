Migraciones — Las 6 tablas en orden
make:migration · Schema · foreignId · cascadeOnDelete · restrictOnDelete
▼

Regla crítica: las tablas padre siempre se crean primero. Si una tabla tiene FK hacia otra, esa otra debe existir ya.

Crear las migraciones en orden

Terminal — en orden, una por una
# 1. Sin FK (tablas padre) php artisan make:migration create_programas_table php artisan make:migration create_instructores_table php artisan make:migration create_estudiantes_table # 2. Con FK (tablas hijas) php artisan make:migration create_fichas_table php artisan make:migration create_matriculas_table php artisan make:migration create_notas_table

1. programas (sin FK)

PHP — create_programas_table
public function up(): void
{
    Schema::create('programas', function (Blueprint $table) {
        $table->id();
        $table->string('nombre', 150);
        $table->string('codigo', 30)->unique();
        $table->unsignedInteger('duracion_horas')->default(2200);
        $table->text('descripcion')->nullable();
        $table->boolean('activo')->default(true);
        $table->timestamps();
    });
}

2. instructores (sin FK)

PHP — create_instructores_table
$table->id();
$table->string('nombre', 150);
$table->string('documento', 20)->unique();
$table->string('correo')->unique();
$table->string('telefono', 20)->nullable();
$table->string('especialidad', 100)->nullable();
$table->boolean('activo')->default(true);
$table->timestamps();

3. estudiantes (sin FK)

PHP — create_estudiantes_table
$table->id(); $table->string('nombre', 150); $table->string('documento', 20)->unique(); $table->string('correo')->nullable(); $table->string('telefono', 20)->nullable(); $table->date('fecha_nacimiento')->nullable(); $table->boolean('activo')->default(true); $table->timestamps();

4. fichas (FK → programas + instructores — RESTRICT)

PHP — create_fichas_table
$table->id(); $table->foreignId('programa_id') ->constrained('programas') ->restrictOnDelete(); // no borrar programa con fichas $table->foreignId('instructor_id') ->constrained('instructores') ->restrictOnDelete(); // no borrar instructor asignado $table->string('numero_ficha', 20)->unique(); $table->date('fecha_inicio'); $table->date('fecha_fin')->nullable(); $table->enum('jornada', ['mañana', 'tarde', 'noche'])->default('mañana'); $table->enum('estado', ['activa', 'terminada', 'cancelada'])->default('activa'); $table->timestamps();

5. matriculas (FK → estudiantes CASCADE / fichas CASCADE)

PHP — create_matriculas_table
$table->id(); $table->foreignId('estudiante_id') ->constrained('estudiantes') ->restrictOnDelete(); // conservar historial del estudiante $table->foreignId('ficha_id') ->constrained('fichas') ->cascadeOnDelete(); // si se borra la ficha, se borra la matrícula $table->date('fecha_matricula'); $table->enum('estado', ['activa', 'retirada', 'finalizada', 'cancelada'])->default('activa'); $table->text('observacion')->nullable(); $table->timestamps();

6. notas (FK → matriculas CASCADE)

PHP — create_notas_table
$table->id(); $table->foreignId('matricula_id') ->constrained('matriculas') ->cascadeOnDelete(); // las notas dependen de la matrícula $table->decimal('nota', 4, 1); $table->string('periodo', 20); // Ej: "2024-1", "2024-2" $table->text('observacion')->nullable(); $table->timestamps();

class Programa extends Model {
    protected $fillable = [
        'nombre', 'codigo', 'duracion_horas', 'descripcion', 'activo'
    ];

    protected function casts(): array {
        return ['activo' => 'boolean'];
    }

        return $this->hasMany(Ficha::class);

Ficha.php (con dos BelongsTo)

PHP — app/Models/Ficha.php
class Ficha extends Model {
    protected $fillable = [
        'programa_id', 'instructor_id', 'numero_ficha',
        'fecha_inicio', 'fecha_fin', 'jornada', 'estado'
    ];
    protected function casts(): array {
        return [
            'fecha_inicio' => 'date',
            'fecha_fin'    => 'date',
        ];
    }
    // Relaciones hacia los padres
    public function programa() {
        return $this->belongsTo(Programa::class);
    }
    public function instructor() {
        return $this->belongsTo(Instructor::class);
    }
    // Relación hacia los hijos
    public function matriculas() {
        return $this->hasMany(Matricula::class);
    }
}

Matricula.php y Nota.php

}

Relaciones en el panel — HasMany y detalle
HasMany · hideOnIndex · detalle del registro · matriculas en ficha
▼

Además de los BelongsTo en los formularios, podemos mostrar los registros hijos directamente en la vista de detalle del padre. Esto es clave para ver las matrículas de una ficha sin salir de ella.

PHP — HasMany en FichaResource (vista detalle)
use MoonShine\UI\Fields\Relationships\HasMany; // Agregar dentro de formFields() al final: HasMany::make( 'Matrículas', 'matriculas', static fn() => new MatriculaResource() ) ->hideOnIndex() // solo visible en el detalle de la ficha ->hideOnForm(), // no aparece al crear/editar la ficha // En InstructorResource — ver fichas del instructor: HasMany::make('Fichas Asignadas', 'fichas', static fn() => new FichaResource()) ->hideOnIndex() ->hideOnForm(), // En MatriculaResource — ver notas de la matrícula: HasMany::make('Notas', 'notas', static fn() => new NotaResource()) ->hideOnIndex() ->hideOnForm(),
💡
hideOnIndex() oculta el campo en el listado de registros. hideOnForm() lo oculta en el formulario de crear/editar. Combinados, el HasMany solo aparece en la vista de detalle del registro (cuando haces clic en "Ver").
09
Roles y Policies — 4 niveles de acceso
role enum · migración · withPolicy · make:policy · canView · canCreate
▼

Moonshine 4.x integra el sistema de authorization de Laravel. Cada Resource puede tener una Policy que controla qué puede hacer cada usuario según su rol.

Los 4 roles del sistema

Super Admin
Acceso total sin restricciones. Puede gestionar usuarios del panel y configuración global.
Coordinador
CRUD completo en todos los módulos. No puede gestionar usuarios del panel.
Instructor
Solo puede ver y editar las fichas que tiene asignadas. No puede crear ni eliminar.
Visitante
Solo lectura. Puede ver listados y detalles, pero no crear, editar ni eliminar.

Paso 1 — Agregar columna role a moonshine_users

Terminal
php artisan make:migration add_role_to_moonshine_users_table
PHP — Migración
public function up(): void { Schema::table('moonshine_users', function (Blueprint $table) { $table->enum('role', [ 'super_admin', 'coordinador', 'instructor', 'visitante', ])->default('visitante')->after('email'); }); } public function down(): void { Schema::table('moonshine_users', function (Blueprint $table) { $table->dropColumn('role'); }); }
Terminal
php artisan migrate # Actualizar el rol del admin que creamos php artisan tinker >>> \MoonShine\Models\MoonshineUser::first()->update(['role' => 'super_admin']);

Paso 2 — Crear la Policy

Terminal
# Crear una Policy por Resource (ejemplo con Ficha) php artisan make:policy FichaPolicy
PHP — app/Policies/FichaPolicy.php
use MoonShine\Models\MoonshineUser; use App\Models\Ficha; class FichaPolicy { // Ver el listado public function viewAny(MoonshineUser $user): bool { return true; // todos pueden ver el listado } // Ver un registro individual public function view(MoonshineUser $user, Ficha $ficha): bool { return true; // todos pueden ver el detalle } // Crear nuevo registro public function create(MoonshineUser $user): bool { return in_array($user->role, ['super_admin', 'coordinador']); } // Editar registro existente public function update(MoonshineUser $user, Ficha $ficha): bool { // super_admin y coordinador pueden editar cualquier ficha // instructor solo puede editar sus propias fichas if (in_array($user->role, ['super_admin', 'coordinador'])) { return true; } if ($user->role === 'instructor') { return $ficha->instructor->correo === $user->email; } return false; // visitante no puede editar } // Eliminar registro public function delete(MoonshineUser $user, Ficha $ficha): bool { return $user->role === 'super_admin'; // solo super_admin elimina } }

Paso 3 — Conectar la Policy al Resource

PHP — FichaResource.php
use App\Policies\FichaPolicy; class FichaResource extends ModelResource { protected string $model = Ficha::class; protected string $title = 'Fichas'; // Conectar la policy — Moonshine la usa automáticamente protected string $policy = FichaPolicy::class; // ... resto del Resource }
✅
Repite el proceso para cada Resource: crear la Policy con make:policy NombrePolicy, definir los métodos viewAny, view, create, update, delete según el rol, y conectarla con protected string $policy en el Resource.

Comandos de deploy en orden

Terminal — Railway CLI
# 1. Instalar Railway CLI (una sola vez) npm install -g @railway/cli # 2. Autenticarse railway login # 3. Vincular con el proyecto en Railway railway link # 4. Subir el código railway up # 5. Ejecutar migraciones en producción railway run php artisan migrate --force # 6. Ejecutar seeders (opcional para producción) railway run php artisan db:seed --force # 7. Crear usuario admin de Moonshine en producción railway run php artisan moonshine:user # 8. Publicar assets de Moonshine railway run php artisan moonshine:publish # 9. Cachear para mejorar el rendimiento railway run php artisan config:cache railway run php artisan route:cache railway run php artisan view:cache
⚠️
Railway no persiste el sistema de archivos entre deploys. Si el proyecto necesita subir imágenes desde Moonshine, configura S3 o Cloudinary como disco de almacenamiento antes de ir a producción.
